# 第一批开发任务

本文档用于把 `docs/02-roadmap.md` 里的 `Now` 阶段拆成可直接执行的首批任务。

适用方式：

- 维护者可以按任务编号逐项推进
- 每完成一项，再回写 README、架构文档或 roadmap
- 单个任务尽量控制在一次提交或一次小范围迭代内完成

## 本批次目标

第一批任务不追求新增大量功能，而是先把模板收敛成更稳定的可复用起点。

本批次关注三件事：

1. 统一模板定位
2. 清理遗留内容和无效噪音
3. 为后续业务模板化打下更清晰的工程基线

## 任务清单

详细 roadmap 请参考 [`02-roadmap.md`](./02-roadmap.md)。

### T001 首页定位收敛

目标：让首页和 README 对项目定位表达一致，不再像纯 demo 首页。

当前背景：

- `README.md` 已经把项目定位为 starter template
- `src/pages/HomePage.tsx` 仍然保留较强的“版本功能展示”风格

建议工作：

- 调整首页文案，突出模板定位而不是版本发布说明
- 将“v1.0.0 新功能”这一类块状信息改成“模板能力”或“推荐起步路径”
- 统一首页中的导航入口文案

涉及文件：

- `src/pages/HomePage.tsx`
- `src/i18n/locales/*.ts`

验收标准：

- 首页与 `README.md` 的项目定位一致
- 首次进入项目的用户能明确看出这是模板，而不是单次版本展示页

优先级：P0

### T002 清理遗留入口 `App.tsx`

目标：明确 `src/App.tsx` 的去留，减少模板中的历史遗留。

当前背景：

- 当前主入口实际使用 `src/AppRouter.tsx`
- `src/App.tsx` 更像 Vite 初期示例演化后的残留文件

建议工作：

- 确认 `src/App.tsx` 是否仍需保留
- 如果不再作为模板示例使用，则删除该文件及其配套样式引用
- 若保留，则需要明确它的职责，并在文档中解释用途

涉及文件：

- `src/App.tsx`
- `src/App.css`
- 相关资源引用文件

验收标准：

- 主入口路径清晰，没有用途不明的历史文件
- 文档中不再出现与实际入口不一致的理解成本

优先级：P0

### T003 依赖清点与瘦身

目标：让依赖清单尽量接近当前真实实现，降低模板噪音。

当前背景：

- 依赖较多，部分是否处于主流程使用状态还不够清楚
- 模板型仓库如果长期不清点依赖，后续会增加理解成本

建议工作：

- 逐项检查 `package.json` 中的依赖是否仍被使用
- 标记“核心依赖”“演示依赖”“可移除依赖”
- 清理明确未使用的依赖
- 在 README 或架构文档中保留核心依赖说明

涉及文件：

- `package.json`
- `package-lock.json`
- 相关代码引用文件

验收标准：

- 依赖项和当前实现基本一致
- 不再保留明显无用途的依赖
- 依赖调整后构建、lint、测试可正常执行

优先级：P0

### T004 环境变量样例补齐

目标：让模板使用者拿到仓库后可以直接完成本地配置。

当前背景：

- 文档已经说明了 `VITE_API_BASE_URL`
- 仓库中还缺少显式的环境变量样例文件

建议工作：

- 新增 `.env.example`
- 补充最小可用的变量说明
- 让 README 中的环境变量说明与样例文件一致

涉及文件：

- `.env.example`
- `README.md`

验收标准：

- 新成员可以直接复制环境变量样例开始开发
- 文档和样例文件保持一致

优先级：P0

### T005 组件示例页与组件文档页表达统一

目标：减少 `/components` 和 `/components/docs` 的信息重复与表达割裂。

当前背景：

- 已经有组件示例页和组件文档页
- 两类页面都在表达组件能力，但侧重点还不够稳定

建议工作：

- 明确 `/components` 只负责交互演示
- 明确 `/components/docs` 只负责 API、用法和代码片段
- 调整两个页面的标题、介绍文案和跳转关系

涉及文件：

- `src/pages/ComponentsPage.tsx`
- `src/pages/ComponentsDocsPage.tsx`

验收标准：

- 两个页面职责清晰
- 用户可以区分“看效果”和“看说明”两个入口

优先级：P1

### T006 API 示例页模板化

目标：把 `/api-demo` 从“能力展示页”进一步整理成“模板接入参考页”。

当前背景：

- `src/pages/ApiDemoPage.tsx` 已经提供 API、icons、hooks 示例
- 但页面表达仍偏 demo 展示，不够像业务模板参考实现

建议工作：

- 调整页面文案，把重点从“展示功能”改成“如何接入”
- 增加更接近真实业务的请求示例说明
- 统一 API 层、hooks 层和页面层的说明关系

涉及文件：

- `src/pages/ApiDemoPage.tsx`
- `src/lib/api.ts`
- `src/lib/useApiQuery.ts`

验收标准：

- 新使用者可以从该页面理解模板中的接口接入方式
- 页面描述和 README / 架构文档口径一致

优先级：P1

### T007 测试基线补强

目标：补上一组最有代表性的模板级测试，提升后续改造安全性。

当前背景：

- 当前测试主要集中在工具函数和 loading hooks
- 首页、主题、i18n、路由等模板核心能力还缺少代表性测试

建议工作：

- 增加首页渲染测试
- 增加主题切换逻辑测试
- 增加路由基础渲染测试
- 增加 i18n 初始化或语言切换的基础测试

涉及文件：

- `src/test/*`
- 相关页面和状态文件

验收标准：

- 模板核心能力至少有一组代表性自动化测试
- 后续清理和重构时能获得基础回归保护

优先级：P1

## 推荐执行顺序

建议按下面顺序推进，避免重复返工：

1. T001 首页定位收敛
2. T002 清理遗留入口 `App.tsx`
3. T004 环境变量样例补齐
4. T003 依赖清点与瘦身
5. T005 组件示例页与组件文档页表达统一
6. T006 API 示例页模板化
7. T007 测试基线补强

## 分批交付建议

如果你希望按小步提交推进，建议拆成三轮：

### 第一轮

- T001
- T002
- T004

目标：先把模板入口和基础接入体验理顺。

### 第二轮

- T003
- T005
- T006

目标：开始清理仓库噪音，并统一演示页面的定位。

### 第三轮

- T007

目标：为后续更大范围调整补上最基本的自动化保护。

## 完成标记建议

后续执行时，建议在本文件中用以下状态维护任务：

- `todo`: 未开始
- `doing`: 进行中
- `done`: 已完成
- `blocked`: 有阻塞

如果你希望继续细化，下一步可以直接把 T001 到 T004 再拆成更具体的代码任务清单。
