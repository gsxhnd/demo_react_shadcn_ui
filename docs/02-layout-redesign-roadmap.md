# 页面排版重构 Roadmap

## 目标

基于现有 React + shadcn/ui + Tailwind + i18n 项目模板，重新实现页面排版与信息架构，建立统一的站点级布局能力，使首页、组件页、文档页、API 页与 404 页具备一致的视觉框架、清晰的导航关系和可持续扩展的页面结构。

本轮重构优先解决页面层问题，不主动改动现有业务能力、数据层封装和组件基础实现。

## 当前问题

1. 页面层缺少统一布局框架，多个页面各自实现背景、容器、页头和操作区。
2. 语言切换、主题切换、页面标题区在多个页面重复编写，复用性低。
3. 首页信息架构偏演示堆叠，不能很好承担“项目模板入口页”的角色。
4. 组件示例页和文档页内容量较大，但缺少稳定的导航和分区结构。
5. 页面之间的入口关系不够清晰，整体站点感较弱。

## 设计原则

1. 先统一框架，再重排页面。
2. 尽量做最小正确改动，优先复用现有路由和页面文件。
3. 公共能力组件化，避免继续在页面内复制布局代码。
4. 保持现有技术栈不变，继续使用 shadcn/ui、Tailwind 和现有状态管理方案。
5. 第一阶段优先完成排版、导航和信息架构，不强制同步重写所有文案国际化。

## 重构范围

### 包含

1. 站点级公共布局抽象。
2. 顶部导航、语言切换、主题切换等公共操作区统一。
3. 首页排版和模块结构重做。
4. 组件示例页、组件文档页、API 页的页面结构重排。
5. 404 页纳入统一视觉框架。

### 暂不包含

1. API 封装逻辑重写。
2. Redux store 结构调整。
3. 基础 UI 组件 API 改造。
4. 大规模新增业务页面。
5. 全量文案国际化补齐。

## 推荐信息架构

### 顶部导航

1. 品牌名 / 项目名。
2. 主导航：首页、组件示例、组件文档、API 接入。
3. 全局操作：主题切换、语言切换。

### 首页

1. Hero 区：模板定位、核心价值、主操作入口。
2. 能力总览区：工程能力、UI 能力、数据能力。
3. 页面入口区：组件示例、组件文档、API 接入。
4. 技术栈区：React、Vite、TypeScript、Tailwind、shadcn/ui、React Query、i18n。
5. 快速开始区：指导用户从模板出发进行扩展。

### 组件示例页

1. 页面标题区。
2. 分类导航区。
3. 分组展示各类组件 demo。
4. 更适合演示和试用，而不是承载完整 API 文档。

### 组件文档页

1. 目录导航区。
2. 组件简介。
3. Props / 子组件说明。
4. 示例代码。
5. 更适合做长期维护的文档页。

### API 页

1. 数据层能力总览。
2. 请求封装说明。
3. Query hooks 使用示例。
4. 约定与最佳实践。

### 404 页

1. 保持统一站点风格。
2. 提供返回首页和返回上一页操作。

## 实施步骤

### 阶段 1：抽离公共布局

目标：建立页面统一骨架，消除重复布局代码。

建议新增：

1. `src/components/layout/AppShell.tsx`
2. `src/components/layout/SiteHeader.tsx`
3. `src/components/layout/PageHero.tsx`
4. `src/components/layout/PageSection.tsx`
5. `src/components/layout/LanguageSwitcher.tsx`

输出：

1. 页面统一背景和容器宽度。
2. 页面统一顶部导航。
3. 主题切换和语言切换从页面内部抽离。

### 阶段 2：接入路由与页面外壳

目标：让所有主页面运行在一致的站点布局内。

执行项：

1. 在路由层引入 layout route，或在页面层统一包裹 `AppShell`。
2. 首页、组件页、文档页、API 页接入同一套头部和内容容器。
3. 404 页保留独立语义，但样式语言与主站统一。

输出：

1. 页面切换时视觉框架一致。
2. 页面间导航关系清晰。

### 阶段 3：重做首页信息架构

目标：把首页从演示卡片集合改造成模板门户页。

执行项：

1. 用 Hero 区明确模板定位。
2. 精简重复卡片，改为结构更清晰的分区展示。
3. 补充对组件页、文档页、API 页的直接入口。
4. 保留当前主题切换、用户态展示等模板能力，但不再让其主导首页布局。

输出：

1. 首页承担项目入口和模板说明角色。
2. 首页更适合后续二次开发和品牌扩展。

### 阶段 4：重排内容页

目标：让内容页具备可浏览、可扩展、可维护的结构。

执行项：

1. `ComponentsPage` 调整为分类浏览式页面。
2. `ComponentsDocsPage` 增加目录化结构，降低长页面阅读成本。
3. `ApiDemoPage` 明确为“数据接入说明页”，突出请求层和 hooks。

输出：

1. 组件演示页更偏展示。
2. 文档页更偏查阅。
3. API 页更偏接入说明。

### 阶段 5：统一收尾与验证

目标：确保页面改版后的整体一致性和可用性。

执行项：

1. 检查桌面和移动端布局表现。
2. 检查深色模式和浅色模式下的视觉一致性。
3. 检查路由跳转、按钮入口、404 行为是否正常。
4. 运行 lint、test、build 做基础验证。

输出：

1. 页面结构稳定。
2. 改版具备可交付状态。

## 风险与注意事项

1. 当前工作区存在未提交变更，实施时需要避免误覆盖已有修改。
2. `ComponentsPage` 和 `ComponentsDocsPage` 内容较长，重构时建议优先处理结构，不一次性重写所有细节。
3. 语言切换与主题切换应统一组件化，否则后续页面继续扩张时重复问题会再次出现。
4. 若后续需要把站点继续演化为文档站或后台模板，当前布局层需要保持中性和可扩展。

## 第一版交付建议

第一版建议以“混合型模板站”为目标，即：

1. 首页具备官网式入口感。
2. 内容页具备文档站式分区结构。
3. 整体仍保留模板演示属性，便于后续二次开发。

## 验收标准

1. 至少抽出一层公共页面布局，不再在每个页面中重复实现头部和操作区。
2. 首页、组件页、文档页、API 页、404 页具备统一视觉语言。
3. 顶部导航可达所有主页面。
4. 移动端和桌面端都能正常浏览。
5. 主题切换和语言切换在全站表现一致。
6. 现有 demo 能力仍可正常访问。
