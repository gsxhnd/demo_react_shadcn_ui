# 开发路线图

本文档只描述后续迭代方向，不再承担历史更新日志职责。

## 当前阶段

项目当前已经具备一个 React starter template 的基础能力，但整体形态仍介于“能力展示 demo”和“可长期复用模板”之间。

接下来的目标不是继续堆积零散功能，而是把已有能力收敛成一套更清晰、更容易复用的模板约定。

## 路线原则

后续迭代遵循以下原则：

- 优先提升模板可复用性，而不是单独增加展示页数量
- 先补齐文档、结构和约定，再扩展业务示例
- 已实现能力与规划能力严格分开
- 每一阶段都要给出可验证的完成标准

## Now

当前优先级最高的工作。

### 1. 文档体系重建

目标：让模板使用者和维护者都能快速理解项目边界。

范围：

- 重写仓库入口文档
- 重写架构说明
- 重做 roadmap
- 明确模板定位、目录职责和推荐开发路径

完成标准：

- 新成员可以仅通过文档完成本地启动
- 文档内容与当前代码结构一致
- 文档中不再出现明显的理想架构冒充现状

### 2. 模板定位统一

目标：把当前项目从“技术栈 demo 集合”收敛为“可直接继续开发的模板”。

范围：

- 统一首页、文档和仓库描述口径
- 明确哪些页面是能力展示页，哪些属于长期模板能力
- 清点遗留示例代码

完成标准：

- README、首页和 roadmap 对项目定位描述一致
- 遗留入口和示例文件有明确保留或清理结论

### 3. 工程收敛与清理

目标：减少模板中的噪音，提高可维护性。

范围：

- 清理未使用或弱使用的依赖
- 检查遗留示例文件
- 补齐基础环境变量说明
- 统一组件示例、API 示例和文档页表达方式

完成标准：

- 项目结构更聚焦
- 依赖清单和当前实现基本一致
- 新使用者不会被默认 demo 残留内容误导

## Next

文档和基础结构稳定后进入的阶段。

### 1. 建立业务模块约定

目标：为项目从模板阶段进入业务开发阶段做准备。

范围：

- 明确是否引入 `features/` 或 `modules/` 目录
- 规定页面、组件、hooks、services 的落位方式
- 给出新增模块的参考结构

完成标准：

- 新业务模块有统一目录约定
- 模板不会因页面增多而迅速失控

### 2. 补齐常见业务模板示例

目标：让仓库从基础设施模板，进一步变成可直接落地的业务 starter。

建议优先增加：

- 表单页模板
- 列表页模板
- 详情页模板
- 空状态、错误态、加载态范式
- 分页与筛选示例

完成标准：

- 使用者可以直接复制基础业务页面结构继续开发
- 示例不只是组件陈列，而是接近真实业务场景

### 3. 鉴权与权限骨架

目标：补齐多数后台型项目的基础能力。

范围：

- 登录态存储策略
- 路由守卫
- 基础权限控制方式
- 未登录和无权限页面处理

完成标准：

- 模板具备最小可用的鉴权接入点
- 路由层和请求层都能接入认证信息

### 4. 测试能力增强

目标：让模板升级为可持续维护的工程基线。

范围：

- 补充页面级和组件级测试示例
- 为路由、主题、i18n 和 API hooks 增加代表性测试
- 明确测试目录和命名约定

完成标准：

- 新增功能有明确测试参考写法
- 测试不再只覆盖工具函数

## Later

中长期方向，用于支撑模板的团队化和产品化使用。

### 1. Storybook 或独立组件文档站

目标：把组件文档从页面内静态展示，演进为更稳定的文档能力。

### 2. E2E 测试

目标：覆盖关键用户路径，降低模板演进时的回归风险。

### 3. CI 工作流

目标：自动执行 lint、test、build，建立基础质量门槛。

### 4. 模板发布与使用指南

目标：让仓库可以被团队内部或外部稳定复用，而不是依赖口口相传。

### 5. 进阶工程能力

可根据需要逐步评估：

- Bundle 分析
- 性能监控
- 错误上报
- SSR 或 SSG 方向
- PWA 能力

## 版本节奏建议

为避免 roadmap 再次演变成历史流水账，建议后续按阶段命名：

| 阶段   | 目标                   |
| ------ | ---------------------- |
| `v1.1` | 文档重建与模板定位统一 |
| `v1.2` | 工程收敛与目录约定稳定 |
| `v1.3` | 业务模板示例与鉴权骨架 |
| `v1.4` | 测试、CI 和文档站完善  |

## 不进入当前阶段的事项

以下内容暂不建议优先推进：

- 过早设计复杂模块体系
- 在没有真实业务约束时提前引入大型抽象层
- 为了“看起来完整”而接入过多工具链

当前最重要的仍然是把模板的边界、约定和使用方式打磨清楚。
